# Hosting the core API docs on Cloudflare Pages

The `@mermaid-lint/core` API reference is generated by [typedoc](https://typedoc.org)
and published to **https://docs.mermaidlint.com**.

Generation and deployment are automated: the [`docs.yml`](../.github/workflows/docs.yml)
workflow runs on every push to `main`, builds the docs, and deploys them with
`wrangler pages deploy`. The build is also exercised on every PR by the CI
`API docs build check (typedoc)` step, so broken doc comments fail review.

This matches how [jasonworden.com](https://jasonworden.com) is deployed
(`wrangler pages deploy` to a Cloudflare Pages project).

## One-time setup

These steps happen once, in the Cloudflare dashboard / GitHub repo settings —
they can't be done from a PR.

1. **Create the Pages project.** In the Cloudflare dashboard → Workers & Pages →
   Create → Pages → "Direct Upload", name it **`mermaid-lint-docs`**. (The name
   must match `--project-name mermaid-lint-docs` in `docs.yml`. The first real
   deploy comes from the workflow; the dashboard step just registers the project.)

2. **Add the repo secrets.** In GitHub → repo Settings → Secrets and variables →
   Actions, add:
   - `CLOUDFLARE_API_TOKEN` — a token with the **Cloudflare Pages: Edit** permission
   - `CLOUDFLARE_ACCOUNT_ID` — the account ID shown in the dashboard URL

   (If these already exist as organization-level secrets, no per-repo action is needed.)

3. **Attach the custom domain.** Pages project → Custom domains → set up
   **`docs.mermaidlint.com`**. Cloudflare adds the CNAME automatically since the
   `mermaidlint.com` zone is on this account.

After that, every push to `main` redeploys the docs. Trigger a manual run any
time from the Actions tab ("docs" workflow → Run workflow).

## Gotcha: the `functions/` directory is reserved

Cloudflare Pages treats a **top-level `functions/` directory** as
[Pages Functions](https://developers.cloudflare.com/pages/functions/) (server
code), and a top-level `_worker.js` as an advanced-mode Worker. Either makes
`wrangler pages deploy` **drop those paths from the static upload** — the pages
then 404 and fall back to the unstyled SPA index.

TypeDoc's default `kind` router emits members into per-kind folders
(`functions/`, `classes/`, …), so `functions/` lands at the deploy root and
breaks every function page. We therefore set **`"router": "structure"`** in
[`packages/core/typedoc.json`](../packages/core/typedoc.json), which uses a flat
layout (`checkSemantics.html`, etc.) with no reserved directory.

[`scripts/check-docs-cloudflare-safe.mjs`](../scripts/check-docs-cloudflare-safe.mjs)
guards this — it runs in both CI (`ci.yml`) and the deploy workflow (`docs.yml`)
and fails if a reserved path reappears at the docs root. Do not revert the router
without re-checking the live site.

## What is *not* part of this

The apex **`mermaidlint.com`** redirect is handled by a separate Cloudflare
Worker (`mermaidlint-redirect`, defined in the `jasonworden.github.io` repo under
`workers/mermaidlint-redirect/`). It is independent of the docs Pages project and
needs no changes here.

## Local preview

```sh
pnpm --filter @mermaid-lint/core docs   # writes HTML to packages/core/docs/
open packages/core/docs/index.html
```

`packages/core/docs/` is gitignored — it is a build artifact, never committed.
